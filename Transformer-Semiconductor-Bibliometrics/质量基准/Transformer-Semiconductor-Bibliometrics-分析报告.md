## 三、亮点与优势

### 3.1 文档体系完善 ⭐⭐⭐⭐⭐

项目文档层次分明，覆盖完整研究生命周期：

```
docs/          → 设计文档（检索式、数据模型、项目大纲）
reports/       → 分析报告（数据质量、方法论、筛选规则）
outputs/       → 数据产出（CSV、指标报告）
Visual output/ → 可视化图谱（6 张 CiteSpace 图）
paper/         → 论文初稿
```

**亮点**：
- `data/field_dictionary.md` 提供了 WoS 52 字段完整字典，对初学者极具参考价值
- `reports/data_quality.md` 给出了 A 级质量评级，数据清洗流程透明
- `config/query.yaml` 记录了检索式三版本迭代历史，可追溯

### 3.2 数据处理严谨 ⭐⭐⭐⭐

- **双集设计**：全量 643 篇 + DL 纯净集 147 篇，兼顾统计稳健性与主题聚焦
- **PRISMA 流程**：遵循 PRISMA 2020 标准，有流程图文档
- **DL 语义筛选**：识别到 "Transformer" 的歧义问题（深度学习 vs 电力变压器），并进行了二次清洗

### 3.3 分析方法规范 ⭐⭐⭐⭐

- CiteSpace 参数固化（g-index k=25, Pathfinder 剪枝, LLR 聚类）
- 网络质量指标完整（Q=0.8411, S=0.9522）
- Kleinberg 突现检测参数明确（s=2, γ=1）

### 3.4 可视化产出丰富 ⭐⭐⭐⭐

6 张 CiteSpace 图谱 + 独立 HTML 展示页 + 三阶段技术演进解读页

## 四、不足与改进建议

### 4.1 🔴 严重问题（P0 级）

#### (1) 检索式时间范围不一致

**问题**：
- README 写的是 "2017-2026"
- `methodology.md` 写的是 "2015-2025"
- 实际数据时间范围未明确标注

**影响**：学术严谨性受损，读者无法确定真实时间范围

**建议**：
1. 统一所有文档中的时间范围表述
2. 在 `data/README.md` 中明确标注实际数据的起止年份
3. 检索式文档中补充时间限制语句（如 `PY=(2017-2026)`）

#### (2) 突现检测报告存在大量无关术语

**问题**：`keyword_burst_report.md` 中检测到 97 个突现词，但大量是电力电子术语：
- "doherty", "mmic", "gallium nitride", "dc-dc converter", "impedance matching"
- 这些是电力变压器/射频电路领域的术语，与 DL Transformer 无关

**原因**：突现检测基于全量 643 篇数据，其中 496 篇是电力电子噪声

**建议**：
1. 对 DL 纯净集（147 篇）单独做突现检测
2. 或在报告中明确区分 "全量集突现" vs "DL 相关突现"
3. 当前报告会误导读者认为这些是 AI 领域的趋势

#### (3) 未使用模板的交互式可视化

**问题**：老师提供的 `bibliometrics-mini` 模板能生成：
- 交互式 Plotly HTML 图谱（可拖拽、缩放、悬停查看指标）
- 综合报告 `bibliometrics_report.html`

但项目只用了模板的数据处理部分，未生成这些核心产出。

**影响**：与课程案例（L17-L18）展示的标准产出不一致

**建议**：
1. 运行模板的完整 pipeline：`python -m bmmini.pipeline --config config/query.yaml --use-wos`
2. 将生成的交互式图谱集成到展示页
3. 或在 README 中说明为何未使用模板可视化

#### (4) 论文初稿方法论部分时间范围错误

**问题**：`manuscript_v1.md` 第 2.1 节写的是 "时间范围 2017-2026"，但 methodology.md 又出现 "2015-2025"

**建议**：全文统一核查数据口径

### 4.2 🟡 中等问题（P1 级）

#### (5) 缺少可复现的 CiteSpace 操作指南

**问题**：
- `baseline/params.md` 只记录了参数，没有操作步骤
- `CiteSpace分析套件/` 目录存在但内容未在 README 中说明
- L17-L18 案例讲解强调了操作可复现的重要性

**建议**：
1. 补充 CiteSpace 分步操作截图指南
2. 或录制一个 5 分钟操作演示视频
3. 明确 `CiteSpace分析套件/` 的内容结构

#### (6) 代码复现性不足

**问题**：
- `requirements.txt` 存在，但未指定版本号
- `run_pipeline.py` 和 `run_pipeline.js` 两个入口并存，职责不清
- `src/` 下有 `.py` 和 `.js` 混用，缺乏统一说明

**对比模板**：模板有清晰的 Makefile 命令和单一入口

**建议**：
1. `requirements.txt` 补充版本锁定（如 `pandas==2.0.0`）
2. README 中明确 Python vs Node.js 的分工
3. 或统一技术栈，删除冗余入口

#### (7) 网络指标计算未完全对齐模板

**问题**：模板定义的指标包括：
- 节点级：degree, weighted_degree, betweenness, pagerank, closeness, eigenvector, community
- 图级：density, n_components, largest_component_ratio, **diameter**, **avg_path_length**

项目的 `network_metrics_*.csv` 是否包含全部这些指标？需核对。

**建议**：对照模板 `src/bmmini/metrics.py` 核查指标完整性

### 4.3 🟢 轻微问题（P2 级）

#### (8) README 结构可优化

**问题**：README 较长（约 300 行），核心发现被埋在中间

**建议**：

3. 考虑拆分为 `README.md`（简洁版）+ `docs/full_report.md`（详细版）

#### (9) 致谢部分 AI 工具罗列过多

**问题**：致谢部分列了 7 个 AI 工具，但未说明各自具体贡献

**建议**：精简为实际主要使用的 2-3 个工具，并说明具体用途

#### (10) 缺少英文版 README

**问题**：项目面向国际学术社区，但 README 仅中文

**建议**：增加 `README_EN.md` 英文版，至少包含：
- 项目简介
- 数据来源
- 核心发现
- 如何复现

#### (11) 仓库体积偏大

**问题**：96 MB 仓库，包含大量 PNG 图片和可能的重复文件

**建议**：
1. 使用 Git LFS 管理大文件
2. 清理 `工作底稿/` 等过程文件
3. 图片压缩后再提交
